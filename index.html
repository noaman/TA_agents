How to read this document: Each BRD process section (2.1–2.8) is mapped to the most appropriate AI automation pattern. The mapping identifies: (1) which process steps can be fully automated, (2) which require an AI agent acting autonomously with tool use, (3) which require multi-agent coordination, and (4) where human approval gates must remain. The goal is to augment—not replace—human judgment for consequential decisions while maximising efficiency on repetitive, rules-based, or information-intensive tasks.

BRD Section 2.1 — Vacancy Request and Approval Workflow. This process is highly amenable to a single intelligent agent that can pre-validate submissions, route intelligently, chase approvers, and surface anomalies — while keeping all approval decisions firmly with humans. The agent operates as an autonomous workflow manager with read/write access to the vacancy record, calendar systems, and notification services.

BRD Section 2.2 — Candidate Sourcing and Shortlisting. This is the highest-value AI automation opportunity in the entire BRD. Each operates with defined tools and scoped responsibilities. Human judgment is preserved at the final shortlist approval gate.

BRD Sections 2.4 & 2.5 — Offer Management, Contract Issuance and Acceptance. The offer workflow combines a single Offer Intelligence Agent (for data validation, comparison table generation, and routing) with automated contract generation. Critical financial decisions remain firmly with human approvers (OD, CPO). The agent's primary value is in eliminating data entry, enforcing validation rules (e.g., 30% salary threshold), and generating the CPO comparison view automatically.

BRD Section 2.6 — Internal Collaboration for Onboarding Tasks. This is the most complex multi-agent use case in the BRD. An Onboarding Orchestrator Agent manages 5 specialist sub-agents (one per internal team: IT, BS, L&T, SS, HRBP) plus a Buddy Assignment Agent. Each sub-agent monitors its team's task list, chases overdue items, escalates per the escalation matrix, and reports status up to the Orchestrator. The result is a fully autonomous onboarding coordination layer requiring human action only on actual task completion.

🔒 Human-in-the-Loop Design Principles

All consequential decisions require human confirmation. Agents prepare, validate, and present — humans decide.

Mandatory Human Gates (12 total)
HRBP Vacancy Approval · OD Vacancy Approval · TA Director Assignment · HM Sourcing Activation · HM Shortlist Accept/Reject · Interviewer Feedback · OD Offer Review · CPO Offer Approval · CPO/HR GM Contract Signature · Recruiter Onboarding Initiation · TA Director Invoice Approval · Finance Invoice Approval
Agent Boundaries
Agents cannot: approve vacancies, make hiring decisions, approve offers, sign contracts, or modify compensation structures. They can: prepare, validate, route, chase, escalate, generate drafts, and update status.

🔧 Shared Agent Infrastructure

Common capabilities consumed by all agents across the system.

Tool Layer (Shared)

Security & Compliance Layer
All agent actions are scoped by RBAC roles (§3.2). Agents cannot access data outside their defined scope. All agent actions are written to audit trail. Sensitive data fields are never passed to LLM prompts in raw form.

⚙️ Guiding Principles for Agentic AI in this System

🛡️

Human Authority on Consequential Decisions

All 12 identified human gates are non-negotiable. Agents prepare and present; humans decide on hiring, approvals, and offers. No agent has authority to autonomously approve a vacancy, select a candidate, or approve an offer.

🔍

Explainability & Audit Trail

Every agent action is logged to the audit trail (BRD §5.4). Screening scores must cite evidence from CVs. Routing decisions must reference the rule applied. The system can explain any automated action taken.

🌐

Entity & Country Awareness

All agents respect the BRD's multi-entity/multi-country requirement. Workflows, checklists, templates, and routing rules are dynamically selected based on the candidate's assigned company and location. No hardcoded single-entity logic.

🔒

Data Minimisation in AI Prompts

Sensitive data (salary figures, personal IDs, EEO data) is never included raw in LLM prompts. Agents use anonymised or tokenised references where AI reasoning is needed, with enrichment done post-inference within the secure system boundary.

📈

Progressive Automation

Phase 1 agents focus on high-certainty, rule-based automation. AI reasoning (LLM-powered screening, ranking, summarisation) is introduced in Phase 1 but with human review. Full autonomous flows are enabled only after baseline data validates agent accuracy.

⚙️

Admin Configurability

All agent thresholds, OLA timers, escalation matrices, and notification templates are configurable via the Admin Center (BRD §5.1) without code changes. The TA Director can tune agent behaviour as operational patterns emerge.
